# Amazon Aurora DSQL

Amazon Aurora DSQL is a serverless, distributed SQL service from AWS that
speaks the PostgreSQL wire protocol. Toasty's PostgreSQL driver connects to
it, but DSQL has a substantially restricted DDL surface and IAM-only
authentication, both of which require care when designing your models.

This page is a feasibility / patterns guide. It is **not** a Toasty
limitation — same model definitions targeted at a stock PostgreSQL backend
work without any modification.

## TL;DR for the feasibility question

Toasty 0.6+ works with DSQL when you observe three constraints:

1. **Use `#[auto] id: uuid::Uuid` for primary keys.** Never `#[auto]` on
   integers — DSQL has no sequences and rejects
   `GENERATED ... AS IDENTITY`. UUID v7 is client-side generated, so no
   DB-side sequence is needed.
2. **Don't use `#[derive(toasty::Embed)] enum`.** Toasty maps it to
   `CREATE TYPE … AS ENUM`, which DSQL rejects. Use plain `String` columns
   plus application-side validation.
3. **Connect with a custom Driver that mints fresh IAM tokens per
   `connect()`.** See `references/guide/custom-driver.md` for the pattern.

Everything else — `#[belongs_to]`, `#[has_many]`, `#[has_one]`,
`#[unique]`, `#[index]`, `Vec<scalar>` arrays, `RETURNING`, `ON CONFLICT`,
transactions — works out of the box. In particular, `#[belongs_to]` does
**not** emit `FOREIGN KEY` constraints in Toasty's `CREATE TABLE` (verified
in `toasty-sql/src/stmt/create_table.rs`), so the lack of FK support in
DSQL is a non-issue.

## DSQL's relevant constraints

| Area | Constraint | Toasty impact |
|---|---|---|
| Auth | IAM-only. Password slot must contain a SigV4 token (15-min TTL for `admin`, up to 1-h for non-admin roles). | Need a custom `Driver` that mints a fresh token on every `connect()`. The stock `PostgreSQL` driver bakes the password into the URL at construction. |
| TLS | Required. `sslmode=verify-full` recommended. | Toasty's `tls` feature (rustls + webpki-roots) accepts the Amazon Trust Services chain DSQL uses. |
| Sequences / `SERIAL` | Not supported. | Avoid `#[auto] id: i64`/`u64`. Use UUID keys. |
| `FOREIGN KEY` constraints | Not supported. | Non-issue — Toasty never emits FOREIGN KEY in `CREATE TABLE`. |
| `CREATE TYPE` / native enums | Not supported. | Avoid `#[derive(toasty::Embed)] enum`. Model enum-shaped fields as `String`. |
| Views, triggers, stored procedures | Not supported. | Toasty does not use any of these. |
| Concurrency | Optimistic — conflicting transactions fail at COMMIT with SQLSTATE `40001`. | Toasty classifies as `Error::SerializationFailure`. **It does not auto-retry.** Callers must implement retry-on-conflict. |
| Transaction limits | Roughly: ~5 min wall-clock, ~3000 modified rows, ~10k scanned per transaction. | Batch jobs need chunking. Large `create!` calls with many nested children or cascading deletes can blow through these. |
| DDL transactionality | Restricted — typically one schema-change operation per transaction. | `db.push_schema()` may work (sends statements individually), but the migration tooling's multi-statement migration files may need splitting. **Untested edge.** |
| `CREATE DATABASE` / `DROP DATABASE` | Not allowed. | `Driver::reset_db()` cannot be implemented; return an error. Use cluster recreation for full reset. |
| `CREATE INDEX` | Async — index becomes ready after `CREATE INDEX` returns. | Code that creates schema then immediately queries the new index may get a sequential-scan plan or be rejected. |

## Model attribute → emitted DDL, with DSQL verdict

This is a quick reference for "is my Toasty schema going to compile on
DSQL?" — derived from reading `toasty-sql/src/serializer/column_def.rs`,
`toasty-sql/src/stmt/create_table.rs`, and `toasty-core/src/driver/capability.rs`.

| Attribute / type | What PG DDL is emitted | DSQL? |
|---|---|---|
| `#[key] #[auto] id: uuid::Uuid` | `"id" UUID NOT NULL`, PRIMARY KEY | ✅ |
| `#[key] #[auto] id: i64` / `u64` / `i32` | `BIGINT GENERATED BY DEFAULT AS IDENTITY` etc. | ❌ no sequences |
| `#[key] id: String` (no `#[auto]`) | `"id" TEXT NOT NULL`, PRIMARY KEY | ✅ |
| `#[unique] field: T` | column + `CREATE UNIQUE INDEX` | ✅ |
| `#[index] field: T` | column + `CREATE INDEX` | ✅ (index build is async) |
| `#[belongs_to(...)]` | foreign-key column only, **no `FOREIGN KEY` constraint** | ✅ |
| `#[has_many]`, `#[has_one]` | no column, no DDL | ✅ |
| `#[derive(toasty::Embed)] struct` | one column per field of the embedded struct | ✅ |
| `#[derive(toasty::Embed)] enum` | `CREATE TYPE foo AS ENUM (...)` + column of that type | ❌ no `CREATE TYPE` |
| `Vec<scalar>` field | native array column (`text[]`, `int8[]`, …) | ✅ |
| `bool`, integers, `String`, `f32/64`, `uuid::Uuid`, `jiff::*`, `rust_decimal::Decimal` | per `references/guide/postgresql.md` type table | ✅ |

## Connecting Toasty to DSQL

There is no first-class DSQL driver in Toasty. The pattern is to wrap
`toasty_driver_postgresql::PostgreSQL` so each new connect mints a fresh
IAM token. The full implementation lives in
`references/guide/custom-driver.md`; the moving parts that are
DSQL-specific:

```rust,ignore
// In Driver::connect(), per call:
let token = aws_sdk_dsql::auth_token::AuthTokenGenerator::new(
    aws_sdk_dsql::auth_token::ConfigBuilder::default()
        .hostname(&hostname)
        .region(aws_sdk_dsql::config::Region::new(region.clone()))
        .expires_in(900)
        .build()?,
)
.db_connect_admin_auth_token(&sdk_config)   // or db_connect_auth_token for non-admin
.await?;
```

The token is the **password** in a standard `postgresql://...` URL.
Percent-encode it (`url::form_urlencoded::byte_serialize`) so the `?`, `&`,
`=`, `+` it contains survive URL parsing. Username is `admin` (or a custom
DB role you've created with `dsql:DbConnect` IAM permission), database name
is `postgres`, port is 5432, `sslmode=verify-full`.

### IAM permission

The principal running the app needs:

- `dsql:DbConnectAdmin` on the cluster ARN, for `admin`-role connects
  (token TTL up to 900 s).
- `dsql:DbConnect` on the cluster ARN, for non-admin role connects
  (token TTL up to 3600 s).

Missing this permission surfaces as a Postgres auth error at connect time,
not at token mint — token minting is purely local HMAC. Easy to
misdiagnose.

### Prefer non-admin roles in production

The 900-second admin token TTL means the pool re-mints frequently. Create a
DB role inside DSQL, grant the schema permissions it needs, and use
`db_connect_auth_token` instead — TTL extends to 1 hour, so 4× fewer
mints.

## Schema lifecycle

**For prototypes / tests: `db.push_schema()`.** Issues `CREATE TABLE` and
`CREATE INDEX` per table, individually. `push_schema` is not idempotent
(it does not use `IF NOT EXISTS`), so the second run errors against an
existing schema. Fine for ephemeral CI clusters; not fine for long-lived
ones.

**For production: the migration system.** See
`references/guide/schema-management.md`. Each migration file is wrapped in
`BEGIN ... COMMIT` by the apply tooling. DSQL restricts DDL transactions
(typically one schema-change op per transaction), so multi-statement
migrations may need to be split into multiple files. This is the largest
unverified-against-DSQL edge in Toasty's migration tooling — review
generated `.sql` files before applying.

## Runtime behavior to handle in application code

**Retry on `SerializationFailure`.** DSQL is OCC — under contention, any
transaction can fail at COMMIT. Toasty surfaces this as
`Error::SerializationFailure` (mapped from SQLSTATE `40001` in
`toasty-driver-postgresql/src/lib.rs`'s `classify_pg_error`). Toasty does
**not** auto-retry. Wrap write paths in an exponential-backoff retry on
that variant.

**Chunk large work.** A single transaction cannot modify ~3000 rows or
run for ~5 minutes. Bulk loads, mass `delete()` cascades, and large
`create!` calls with many nested children are the typical offenders.
Batch them.

**Wait briefly after `CREATE INDEX`.** Indexes build asynchronously.
Workloads that bootstrap the schema and immediately query a new index can
get rejected plans until the index is online. Sleep 1-2 seconds, or poll
`pg_index.indisready`.

## Dev / test workflow

DSQL has no local emulator. The pragmatic pattern:

- **Inner loop:** run a local Postgres in Docker (`postgres:17` is fine)
  and connect Toasty to it via the stock `PostgreSQL` driver. Schemas that
  follow the constraints in the table above will work identically.
- **CI:** keep a long-lived "ci" DSQL cluster. Prefix table names per run
  (`db.builder().table_name_prefix("run_abc_")`) so concurrent CI jobs
  don't collide. Clean up table prefixes at the end. A fresh DSQL cluster
  per test is too slow (~5 minutes to provision).
- **Production:** real DSQL cluster, multi-region if you need that.
